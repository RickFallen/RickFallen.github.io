起因

公司有次做活动涉及到扣库存,由于没有采取任何加锁措施,结果显而易见,奖品超发,悲剧…

乐观锁介绍

  • what

    乐观锁是一种并发控制策略,并不是数据库提供的机制,也就意味着需要我们从sql层面上手动实现这种机制.要实现乐观锁需要在表中添加一个version字段(或者时间戳,hash,或者是要扣减的字段本身等等),在事务开始时首先执行查询操作记录version的值,然后执行修改操作,然后校验此时version的值是否跟之前查询的值相等,如果相等,提交事务,否则进行回滚,重试等操作.

  • when

    乐观锁用于典型的并发扣减库存场景,当一条update语句执行时,需要同时满足一个或多个条件才能执行成功,可以认为是幂等性的实现,既多次执行结果始终相同.

  • why

    为什么使用锁,是因为涉及到多个线程同时操作同一个资源,也就是出现所谓的竞态条件,先查询后更新并不是原子性操作,会导致数据不一致,因此就需要锁,使得先查询后更新是一个原子性操作.

    为什么使用乐观锁,是因为乐观锁适用于高并发场景,当使用悲观锁时可能会导致大量线程处于等待状态,系统吞吐量下降,用悲观锁必须要小心,因为还有可能出现死锁.

    但是并不代表悲观锁不好,什么时候使用乐观锁和悲观锁要视情况而定.

    本文主要讲乐观锁,关于悲观锁此处不做展开.

乐观锁实现方式

话不多说直接上伪代码

  • 普通乐观锁

    SELECT iD, val1, val2
     FROM theTable
     WHERE iD = @theId;
#{code that calculates new values}
UPDATE theTable
     SET val1 = @newVal1,
         val2 = @newVal2
     WHERE iD = @theId
         AND val1 = @oldVal1
         AND val2 = @oldVal2;

  • 基于事务的多字段乐观锁

    要注意数据库的事务隔离级别

    SELECT iD, val1, val2
     FROM theTable
     WHERE iD = @theId;
#{code that calculates new values}
BEGIN TRANSACTION;
UPDATE anotherTable
     SET col1 = @newCol1,
         col2 = @newCol2
     WHERE iD = @theId;
UPDATE theTable
     SET val1 = @newVal1,
         val2 = @newVal2
     WHERE iD = @theId
         AND val1 = @oldVal1
         AND val2 = @oldVal2;
#{if AffectedRows == 1 }
    COMMIT TRANSACTION;
    #{go on with your other code}
#{else}
    ROLLBACK TRANSACTION;
    #{decide what to do since it has gone bad... in your code}
#{endif}

  • 基于事务的version的乐观锁

    同样需要事务,只不过这里省略了

    SELECT iD, val1, val2, version
     FROM theTable
     WHERE iD = @theId;
#{code that calculates new values}
UPDATE theTable
     SET val1 = @newVal1,
         val2 = @newVal2,
         version = version + 1
     WHERE iD = @theId
         AND version = @oldversion;
#{if AffectedRows == 1 }
    #{go on with your other code}
#{else}
    #{decide what to do since it has gone bad... in your code}
#{endif}
